REPORT COMPLETO — Sistema Overflow/Concurrency Alessia AI Voice Platform
Contesto
La piattaforma gestisce chiamate vocali AI (inbound via FreeSWITCH + VPS Bridge, outbound via Telnyx SIP). Il sistema precedente aveva un limite di concurrency globale. In queste sessioni abbiamo implementato un sistema per-numero con 6 miglioramenti interconnessi.

GAP 1: Outbound — Limite per-numero invece di globale
File: server/routes/voice-router.ts, funzione executeOutboundCall (riga ~2516)

Cosa è stato fatto:

La determinazione di sipCallerId (righe 2541–2555) è stata spostata PRIMA del concurrency check. Si usa una gerarchia: call.from_number → users.sip_caller_id → consultant_numbers.phone_number (stato active)
Il concurrency check (righe 2557–2644) ora:
Auto-scade i record calling più vecchi di 5 minuti (riga 2559–2565)
Legge max_concurrent_calls da voice_numbers per il sipCallerId specifico, con fallback a 5 (righe 2567–2578)
Conta le chiamate outbound attive per quel numero specifico (from_number = sipCallerId), non globali (righe 2581–2597)
Se il limite è raggiunto, ri-accoda la chiamata con backoff esponenziale: 3min × 2^attempts, cap 15min (righe 2629–2643)
Il campo max_concurrent_calls è nella tabella voice_numbers (intero, default 5, range 1-10)
La variabile locale ovMaxConcurrent nel frontend (riga 4412) legge n.max_concurrent_calls ?? 5 ed è usata per tutti i salvataggi, evitando sovrascritture stale
Stato precedente rimosso: La query a consultant_availability_settings.max_concurrent_calls per l'outbound è stata sostituita con la query a voice_numbers

GAP 3: Coordinamento outbound+inbound — Conteggio unificato
File: server/routes/voice-router.ts, dentro executeOutboundCall (righe 2599–2627)

Cosa è stato fatto:

Dopo aver contato le outbound attive dal DB (righe 2581–2597), executeOutboundCall fa una HTTP GET al VPS /health (righe 2605–2609) con timeout 3 secondi via AbortController
Dal response, legge activeByNumber e cerca il numero sipCallerId (normalizzando per confronto digit-only, righe 2613–2619)
Somma: totalActiveForNumber = activeScheduledForNumber + vpsActiveForNumber (riga 2626)
Se VPS non risponde (catch a riga 2621), prosegue con solo conteggio locale come fallback — la chiamata non viene bloccata per un VPS irraggiungibile
Il check finale (riga 2629) confronta totalActiveForNumber >= maxConcurrentForNumber
Log dettagliato include sia outbound count che VPS inbound count (riga 2641)
Nota architetturale: activeVoiceResult (voce_calls con status='active', riga 2591) è usato solo per totalActiveGlobal (riga 2627) come safety cap secondario, non per il check per-numero

GAP 5: Dialplan — Timeout audio, messaggio personalizzato, posizione coda
File: vps-voice-bridge/freeswitch/dialplan/default/05_overflow_queue.xml

Extension overflow_queue (riga 42):

Flusso: bind_digit_action 1 → 6 cicli di announce_position + play_announcement + hold_music.wav → park se nessun dequeue
Ciclo singolo = ~1-2 min (annuncio ~15s + musica 30s)
set playback_terminators=none impedisce che DTMF interrompa i playback
Extension announce_position (riga 85):

Condizione: ${queue_position} matcha regex ^[1-9]\d*$ (solo numeri positivi)
Se sì: riproduce position_prefix.wav → say it NUMBER PRONOUNCED ${queue_position} → position_suffix.wav
Es: "Sei in posizione numero... due... nella coda d'attesa"
Extension play_announcement (riga 96):

Attualmente riproduce sempre announcement.wav (il file con DTMF)
La variabile overflow_custom_message viene settata via ESL (riga 328–330 di voice-bridge-server.ts) ma il dialplan non la usa ancora per TTS — usa solo il file WAV statico
Limitazione nota: TTS runtime non implementato nel dialplan. Il overflow_custom_message viene passato come variabile channel ma non letto. Per usarlo servirebbe mod_tts_commandline con Google TTS o simile. Il campo è comunque salvato nel DB e settato via ESL per uso futuro.
Extension overflow_timeout (riga 103):

Triggered quando il Voice Bridge fa uuid_transfer <uuid> overflow_timeout XML default
Clear DTMF bindings → riproduce timeout.wav → hangup NORMAL_CLEARING
Extension transfer_to_fallback (riga 112):

Triggered da DTMF "1" via bind_digit_action
Clear DTMF → transferring.wav → bridge sofia/gateway/${sip_gateway}/${tech_prefix}${fallback_transfer_number}
Se bridge fallisce: transfer_failed.wav → hangup
Variabili channel settate dal Voice Bridge (righe 316–330 di voice-bridge-server.ts):

sip_gateway, tech_prefix: per il bridge SIP verso fallback
queue_position: posizione nella coda (calcolata come overflowCount + 1)
fallback_transfer_number: solo se DTMF abilitato e fallback configurato
overflow_dtmf_enabled: true/false
overflow_custom_message: testo sanitizzato (; e = sostituiti con spazio)
Le variabili vengono settate via uuid_setvar_multi prima del uuid_transfer (100ms di delay per sicurezza)
File: vps-voice-bridge/scripts/generate-overflow-audio.sh

Script a 8 step che genera tutti i WAV necessari:

announcement.wav — messaggio con opzione DTMF
announcement_no_dtmf.wav — messaggio senza DTMF
transferring.wav — "Ti stiamo trasferendo..."
transfer_failed.wav — "Il trasferimento non è riuscito..."
timeout.wav — "Ci scusiamo... riprova più tardi..."
position_prefix.wav — "Sei in posizione numero"
position_suffix.wav — "nella coda d'attesa."
hold_music.wav — 30s di synth (sine 440:880 + 330:660, tremolo, reverb)
TTS engine: espeak-ng con voce italiana (-v it -s 140 -p 50), conversione a 8kHz mono 16-bit via sox. Prerequisiti: sudo apt-get install espeak-ng sox

GAP 6: Aggiornamento dinamico posizione coda dopo dequeue
File: vps-voice-bridge/src/voice-bridge-server.ts, funzione tryDequeueOverflow (righe 657–750)

Come funziona tryDequeueOverflow:

Chiamata quando una sessione termina (endSession → tryDequeueOverflow(), riga 654)
Guard: overflowCount === 0 o dequeueInProgress → exit
Loop while: prende il prossimo dalla coda con getNextOverflow() (FIFO, riga 665)
Calcola timeout residuo: remainingMs = max(5000, originalTimeout - elapsed) (riga 678)
Check per-numero: se il numero del chiamante è ancora pieno → reinsertAtFront() con timeout residuo → break (riga 694–698)
Check globale: se pieno → reinsertAtFront() → break (riga 700–703)
Check auto-return: se disabilitato per quel numero → skip (riga 706–708)
Check uuid_exists: verifica che il chiamante sia ancora in linea (riga 711–716)
Se non esiste: skip, prova il prossimo (riga 718–720)
Transfer: uuid_transfer <uuid> alessia_ai_9999_public XML public (riga 725)
Aggiornamento posizioni (righe 734–741): dopo un dequeue riuscito, per ogni entry rimasta nella coda, aggiorna queue_position via uuid_setvar (1-indexed). Questo è best-effort — il dialplan legge la variabile solo al prossimo ciclo di annuncio, non interrompe l'audio in corso.
Metodo reinsertAtFront (session-manager.ts riga 320): ri-mette l'entry in testa alla coda con un nuovo timeout. Preserva l'ordine FIFO originale e il enqueuedAt originale. Il timeout usa il tempo residuo, non il timeout intero.

LIMITAZIONI NOTE
overflow_custom_message non usato in TTS runtime: il campo è salvato nel DB, passato via ESL come variabile channel overflow_custom_message, ma il dialplan play_announcement riproduce sempre announcement.wav. Per TTS runtime servirebbe mod_tts_commandline con un engine esterno (Google TTS, Amazon Polly). Il campo è pronto per uso futuro.

Posizione coda best-effort: uuid_setvar queue_position aggiorna la variabile, ma il dialplan la legge solo quando raggiunge di nuovo execute_extension announce_position nel loop. Non interrompe l'audio in corso.

VPS health fallback: se il VPS non risponde entro 3 secondi, executeOutboundCall prosegue con solo conteggio locale. Questo significa che in caso di VPS down, il limite per-numero potrebbe non includere le inbound attive.

announcement.wav vs announcement_no_dtmf.wav: il dialplan usa sempre announcement.wav (con DTMF), anche quando overflow_dtmf_enabled=false. Andrebbe aggiunto un branch nel play_announcement extension per usare il file senza DTMF.

Conteggio outbound per-numero: il filtro from_number = sipCallerId nella query (riga 2585) funziona solo se le chiamate hanno from_number valorizzato. Se una chiamata non ha from_number settato, non viene contata per quel numero. Questo è di design — le chiamate senza numero esplicito usano il default del consulente.

espeak-ng qualità audio: la voce sintetica di espeak-ng è riconoscibilmente robotica. Per produzione si consiglia di sostituire i file WAV con registrazioni professionali o output di un TTS cloud (Google TTS, ElevenLabs).

ARCHITETTURA RIEPILOGATIVA
Chiamata INBOUND:
  Telnyx → FreeSWITCH → WebSocket → VPS Voice Bridge
    → sessionManager.canAcceptCallForNumber(num, max)?
      SÌ → createSession() → Gemini AI
      NO → overflow_enabled?
        SÌ → uuid_setvar_multi (queue_position, vars) → uuid_transfer overflow_queue → addToOverflow()
        NO → uuid_kill CALL_REJECTED
    → Quando sessione termina → tryDequeueOverflow()
      → Primo in coda → uuid_transfer back to AI → aggiorna posizioni rimaste
Chiamata OUTBOUND:
  Replit → executeOutboundCall()
    → Determina sipCallerId (gerarchia: call.from → user.sip → consultant_number)
    → Query voice_numbers.max_concurrent_calls per quel numero
    → Conta outbound attive per quel numero (scheduled_voice_calls + voice_calls)
    → Fetch VPS /health → conta inbound attive per quel numero (activeByNumber)
    → totalActive >= maxConcurrent?
      SÌ → Re-queue con backoff esponenziale
      NO → Procedi con la chiamata
UI Consulente:
  → Card "Gestione Overflow": per ogni numero, configurazione overflow con live status
    (pallino verde/rosso, X/Y linee, X in coda con dettagli per chiamante)
  → Card "Coda Chiamate": per-number grouping con barra progresso
  → Polling ogni 10s via /api/voice/health → proxy VPS /health
